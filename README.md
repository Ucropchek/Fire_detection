Для ускорения ручного отбора изображений из перечисленных источников разработано отдельное **Python + tkinter приложение**:
- показывает изображения по одному, одобрение/отклонение одним нажатием (`✓`/`✗` или Enter/Backspace);
- одобренные изображения с разметкой автоматически раскладываются по train/val;
- ведёт статистику по категориям сцен (лес, город, помещение, ночь, дальний/ближний план и др.) — для контроля разнообразия выборки;
- исключает пересечение train/val по имени файла, поддерживает отмену действия и продолжение с места остановки.

## 4. Разметка данных

Формат разметки — bounding boxes (детекция объектов), платформа — **Roboflow**.

- изображения из открытых датасетов — готовая разметка, проверена и скорректирована вручную;
- кадры из видео — размечены с нуля вручную.

**Обнаруженная и исправленная проблема разметки:** при переименовании класса `0` в `fire` через интерфейс Roboflow старые аннотации не пересчитались автоматически — в датасете образовалось два независимых обозначения одного объекта. Исправлено объединением класса `0` в `fire` на Roboflow с генерацией новой версии датасета.

## 5. Освоение фреймворка Ultralytics

Изучены: загрузка предобученной модели, `model.train()`, `model.val()`, `model.predict()`, работа с `data.yaml`, чтение метрик из объекта результатов (`.box.map50`, `.box.map`, `.box.mp`, `.box.mr`).

### Применённые аугментации и обоснование выбора
 
| Параметр | Значение | Обоснование |
|---|---|---|
| `degrees` | 10 (лёгкий поворот) | Имитирует небольшой наклон камеры видеонаблюдения/дрона. 10° — разумный компромисс: не создаёт нереалистичных ракурсов. |
| `translate` | 0.1 (сдвиг 10%) | Огонь редко в центре кадра — сдвиг учит модель искать объект по всему полю зрения, а не только в центре. |
| `scale` | 0.5 (масштаб ±50%) | Критично для fire detection: очаг возгорания в датасете может быть и маленьким (начало пожара, дальний план), и почти во весь кадр (крупный план/близкая камера). Широкий диапазон масштаба — оправдан. |
| `shear` | 2.0 (небольшой сдвиг) | Минимальная перспективная деформация, имитирует неидеальную геометрию установки камеры, влияние небольшое. |
| `perspective` | 0.0005 (почти отключено) | Правильно — сильные перспективные искажения нетипичны для статичных камер, поэтому значение близко к нулю. |
| `flipud` | 0.1 (вертикальный флип, низкая вероятность) | Спорный момент: физически пламя направлено вверх, флип создаёт «перевёрнутый огонь». Низкая вероятность (0.1) снижает вред, но стоит поставить под сомнение — если в датасете нет случаев съёмки под необычным углом (отражения, экраны, нестандартный монтаж камеры), лучше 0.0. Если оставляли специально для устойчивости к переворотам видео — допустимо. |
| `fliplr` | 0.5 (горизонтальный флип) | Полностью оправдан — у пламени нет «правильной» стороны, зеркалирование безопасно и увеличивает разнообразие. |
| `mosaic` | 1.0 (всегда включено) | Помогает модели видеть мелкие/удалённые очаги в контексте разных сцен, повышает recall на маленьких объектах — важно, раз recall низкий (0.38). Но: `close_mosaic` не указан, значит mosaic работает все 150 эпох без отключения в конце — это может мешать финальной точной сходимости геометрии боксов (коррелирует с низким mAP50-95 и большим разрывом с mAP50). Стоит добавить `close_mosaic=10-15`. |
| `mixup` | 0.15 (смешивание изображений) | Спорно для одного класса с характерной полупрозрачной/светящейся текстурой — блендинг двух кадров может создавать нереалистичные «полупрозрачные» артефакты, не похожие на настоящий огонь. Может объяснять просадку precision (0.47) — модель видит «ложный огонь» из смешанных пикселей и учится на шуме. Кандидат на снижение/отключение при повторном эксперименте. |
| `hsv_h` | 0.015 (сдвиг оттенка) | Низкое значение — правильно, чтобы не терять характерный оранжево-жёлтый оттенок пламени. |
| `hsv_s`, `hsv_v` | 0.7, 0.4 (сатурация/яркость) | Довольно агрессивные значения (особенно `hsv_s=0.7` — очень широкий разброс). Обосновано желанием охватить разные условия освещения (дым, сумерки, прожектор), но при 0.7 сатурация может сильно «выцветить» или «перенасытить» цвет пламени — кандидат на смягчение (например, до 0.4–0.5), если нужно поднять precision. |
| `dropout` | 0.1 (регуляризация головы) | Небольшой dropout снижает переобучение классификационной головы — разумно при небольшом датасете. |
| `weight_decay`, `lr0`, `lrf`, `cos_lr` | 0.0005, 0.01, 0.01, True (стандартные значения Ultralytics) | Дефолтные и проверенные настройки, cosine-annealing даёт плавное снижение LR к концу — хорошо согласуется с тем, что лоссы гладко убывают. |

## 6. Обучение модели

Среда: **Google Colab, GPU Tesla T4** (доступ к серверу компании на момент работы отсутствовал).

### Путь к финальной версии

Первая версия обучалась на **YOLO11n** — самой лёгкой модели семейства, как отправная точка для проверки всего пайплайна (скачивание датасета → обучение → валидация). На nano-модели была обнаружена и устранена ключевая проблема — дублирование разметки класса `fire` (см. п.4): после исправления mAP50 выросло почти в 3 раза (с 0.107 до 0.313).

После стабилизации пайплайна модель была заменена на **YOLO11s** — итоговую версию проекта. Small была выбрана вместо nano как более точная архитектура при сохранимой скорости инференса, и вместо medium — из-за риска переобучения на датасете такого размера (~800 изображений) при заметно большем числе параметров medium-модели.

### Технические трудности при настройке окружения (актуально для обеих моделей)

- библиотеки `roboflow`/`ultralytics` приходилось переустанавливать в каждой новой сессии Colab;
- обучение без переключения runtime на GPU вызывало ошибку недоступного CUDA-устройства;
- конфликт версий `sympy` и установленного PyTorch, потребовавший фиксации совместимой версии.

### Итоговые параметры обучения (YOLO11s)

```python
model = YOLO("yolo11s.pt")
model.train(
    data="data.yaml", epochs=150, imgsz=640, batch=16,
    patience=30, device=0,
    degrees=10, translate=0.1, scale=0.5, shear=2.0, perspective=0.0005,
    flipud=0.1, fliplr=0.5, mosaic=1.0, mixup=0.15,
    hsv_h=0.015, hsv_s=0.7, hsv_v=0.4,
    dropout=0.1, weight_decay=0.0005, lr0=0.01, lrf=0.01, cos_lr=True,
)
```

### Полученные визуализации с использованием встроенных средств Ultralytics

#### Confusion matrix

<center><img src="readme_data/confusion_matrix.png"></center>

Модель путает огонь с фоном примерно поровну в обе стороны (~250 ложных срабатываний фон→fire, ~252 пропуска fire→фон) — граница между классами ещё не выработана уверенно, особенно на объектах с тёплым освещением (кожа человека у костра, закат).

#### Results

<center><img src="readme_data/results.png"></center>

**Train-лосcы (box/cls/dfl)** — стабильно и монотонно снижаются на всём протяжении обучения, без плато. Это значит, что модель продолжает учиться.

**Val-лоссы** — резкий выброс в первые ~5 эпох (`val/cls_loss` до 2500, `val/dfl_loss` до 25) — типичная нестабильность на старте (веса ещё не устаканились, LR warmup). После эпохи ~10 всё быстро стабилизируется и держится на плато:

- `val/box_loss` ≈ 2.1
- `val/cls_loss` ≈ 0
- `val/dfl_loss` ≈ 2.5

## 8. Проверка модели на видео

Рассмотрено два подхода к инференсу:

| Подход                                        | Решение                                                                |
| --------------------------------------------- | ---------------------------------------------------------------------- |
| Roboflow Inference Server (Docker + REST API) | отклонён — требует установки Docker и постоянно запущенного контейнера |
| **Собственное Python-приложение (выбрано)**   | использует `best.pt` напрямую через Ultralytics, без сервера и сети    |

Приложение: диалог выбора видеофайла → покадровое чтение через OpenCV → инференс каждого кадра моделью YOLO11s → отрисовка bbox/класса/уверенности поверх видео в реальном времени, без сохранения на диск.

Не требует API-ключа, интернета или дополнительного ПО — переносимо между компьютерами без настройки инфраструктуры.

## Ограничения и направления развития

Основное ограничение — объём датасета (~800/200) и умеренный Recall (0.28–0.43), приводящие к путанице огня с визуально похожими тёплыми/яркими объектами. Дальнейшее развитие: расширение датасета отрицательными примерами, более плотная разметка границ пламени, увеличение `imgsz` при обучении, эксперимент с YOLO11m при пропорциональном росте датасета.
